# Day 3 - ChromaDB and Vector Databases

## Summary

Today I learned about Vector Databases and implemented retrieval using ChromaDB.

In Day 2, retrieval was implemented manually using Python lists and cosine similarity.

Today, ChromaDB automated the process of:

* Storing embeddings
* Managing documents
* Performing similarity search
* Retrieving relevant documents

This is much closer to how production RAG systems operate.

---

# Why Do We Need Vector Databases?

In Day 2, every query required comparing the query embedding against every stored document embedding.

Example:

Query

↓

Document 1

Document 2

Document 3

↓

Cosine Similarity

↓

Best Match

This approach works for a few documents but becomes inefficient when thousands or millions of documents exist.

Vector databases solve this problem by efficiently storing embeddings and retrieving the nearest vectors.

---

# What Is ChromaDB?

ChromaDB is an open-source vector database.

Its primary purpose is:

Documents

↓

Embeddings

↓

Storage

↓

Semantic Search

↓

Relevant Results

It is commonly used in:

* RAG Systems
* AI Chatbots
* Semantic Search
* Knowledge Bases
* LLM Applications

---

# Core Concepts

## Client

The Client represents the connection to ChromaDB.

Example:

client = chromadb.Client()

Think of it as opening a database connection.

---

## Collection

A Collection is similar to a SQL table.

Example:

collection = client.create_collection(
name="internship_docs"
)

A collection stores:

* Documents
* Embeddings
* Metadata
* IDs

---

## Documents

Documents are the actual text stored in the database.

Example:

"Interns receive a stipend of 15000 rupees per month"

---

## IDs

Every document requires a unique identifier.

Example:

ids = [
"doc1",
"doc2",
"doc3"
]

This allows ChromaDB to uniquely identify documents.

---

# Understanding collection.add()

Example:

collection.add(
documents=[...],
ids=[...]
)

Internally:

Document

↓

Embedding Model

↓

Embedding Vector

↓

Stored In Collection

Unlike Day 2, embeddings are automatically generated by ChromaDB.

---

# Understanding collection.query()

Example:

results = collection.query(
query_texts=[
"How much money do interns receive?"
],
n_results=1
)

Internally:

Question

↓

Embedding

↓

Similarity Search

↓

Ranking

↓

Top Result

This is the retrieval step.

# Understanding Results

The query method returns a dictionary.

Example:

{
"documents": [...],
"ids": [...],
"distances": [...]
}

To access retrieved documents:

results["documents"]

To access the first document:

results["documents"][0][0]

# Key Takeaways

1. ChromaDB is a vector database.

2. Vector databases store embeddings instead of traditional rows and columns.

3. Collections are similar to tables.

4. add() stores documents and embeddings.

5. query() performs similarity search.

6. ChromaDB automatically generates embeddings.

7. Vector databases automate the retrieval process built manually in Day 2.

8. ChromaDB is commonly used in production RAG systems.

9. Retrieval quality still depends on document quality and semantic context.

10. ChromaDB is the first real vector database used in this learning journey.

# Interview Questions I Can Answer

What is a vector database?

A system that stores embeddings and performs similarity search.

What is ChromaDB?

An open-source vector database used for semantic retrieval.

What is a Collection?

A container that stores documents, embeddings, metadata, and IDs.

What happens during add()?

Documents are converted into embeddings and stored.

What happens during query()?

The query is embedded and compared against stored embeddings to retrieve the most relevant results.
